# /summit-brainstorm

Tighten the current SummitHarness brief before planning or implementation continues.

## Workflow

1. Confirm `scripts/context_engine.py`, `.codex-loop/workflow/ONBOARDING.md`, `.codex-loop/intake/ANSWERS.md`, `.codex-loop/intake/APPROVAL.md`, `.codex-loop/prd/PRD.md`, and `.codex-loop/prd/SUMMARY.md` exist.
2. Read the workflow onboarding and idea notes first, then read intake answers and approval, then `.codex-loop/prd/PRD.md`, `.codex-loop/prd/SUMMARY.md`, `.codex-loop/PROMPT.md`, and `.codex-loop/context/handoff.md` if present.
3. Tighten the brief so it clearly names the approved user, problem, desired outcome, constraints, first-release boundary, and acceptance bar.
4. If UI matters, capture concrete design direction and state coverage instead of generic polish notes.
5. Record visible assumptions instead of hiding them, and keep unresolved questions visible when approval is still partial.
6. Refresh the context packet if the brief materially changed, then summarize what is now locked and what still needs a decision.

## Notes

- This command is most useful right after intake or when the task graph no longer matches the real goal.
- Prefer sharpening the approved brief over inventing implementation detail too early.
