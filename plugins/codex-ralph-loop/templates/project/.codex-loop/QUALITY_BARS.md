# Quality Bars

The active profile is selected by `.codex-loop/config.json` under `loop.quality_profile`.
If it is missing, map `proposal`, `planning`, `submission`, and `prd` modes to `proposal`; map everything else to `development`.
Completion requires satisfying every `MUST` item in the active profile.

## Development

Use this when the goal is shipping code, fixing behavior, or delivering a runnable product slice.

### MUST
1. Outcome reality
- A meaningful workflow, bugfix, or feature slice works end to end.
- Do not declare completion on scaffolding alone when a runnable slice is possible.

2. Verification evidence
- Relevant tests, checks, or screenshots actually ran and passed.
- If the change is UI-facing, include rendered evidence, not only source code edits.

3. Product quality
- Handle the important runtime states for the changed behavior: success, failure, and any obvious loading/empty state.
- Do not leave critical TODOs, placeholder copy, fake data, or manual steps unless the user explicitly accepted them.

4. Design quality
- If a UI exists, it must feel intentional rather than autogenerated.
- Fail generic card spam, weak hierarchy, sparse layouts, or obviously unfinished polish.

5. Integration truthfulness
- Update contracts, docs, config, migrations, or supporting files when behavior changed.
- Claims in docs and summaries must match the actual repo state.

### FAIL conditions
- Checks were skipped or are weak for the risk level.
- The main user path is still incomplete.
- The UI still looks generic or unfinished.
- Completion is claimed while important gaps are hidden in notes or TODOs.

## Proposal

Use this when the goal is a PRD, pitch, submission package, strategy memo, deck, or other persuasion-first document.

### MUST
1. Problem framing
- Name the target reviewer, buyer, or operator.
- State the problem, why it matters now, and what changes if the idea works.
- Ground the claim with evidence, not just intuition.

2. Current vs future separation
- Distinguish clearly between what the repo already proves and what is still the next-step vision.
- Current implementation should appear as feasibility evidence, not consume the main narrative.

3. Visual communication
- Include every visual, table, flow, evidence capture, and comparison artifact that the reviewer needs to understand the problem, solution, architecture, feasibility, and expected effect without guessing.
- If the narrative depends on a comparison, sequence, workflow, structure, or external signal, show it visually instead of burying it in prose.
- If the problem statement depends on market, policy, regulatory, or news context, include source-backed evidence captures or equivalent visual proof instead of leaving it as unsupported copy.
- Visuals and tables must carry information, not decoration.

4. Page quality and narrative density
- No page should look sparse, placeholder-like, or template-generic.
- Every page should have a distinct narrative job.
- Prefer strong hierarchy, clean composition, and reviewer-friendly density over ornamental styling.

5. Submission readiness
- Provide reusable one-line summary, keywords, solution summary, feasibility, business path, and expected effect.
- Remove vague AI filler, fake metrics, unsupported claims, and empty marketing language.

### FAIL conditions
- Necessary visual explanations, evidence captures, flow diagrams, or comparison tables are missing.
- Problem urgency, feasibility, or differentiation is asserted without the supporting capture, diagram, or comparison structure that would let a reviewer verify it quickly.
- The document is mostly prose with little visual explanation.
- The pages look like autogenerated boxes or thin templates.
- Current implementation and future vision are blurred together.
- The reviewer cannot quickly reuse the document in a form or presentation.
